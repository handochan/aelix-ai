## Decision

**Two-tier split with the LLM tier untouched.** A consumer sweep showed every existing `build_context().messages` consumer (turn-start state rebuild, compaction summarizer, runtime session swap, print-mode) is LLM-tier; only the TUI replay wants rich messages. So instead of pi's rich-context + convert-at-agent-boundary (which would push a new `role:"custom"` through every `_state.messages` heuristic, token estimation, and provider adapter), aelix keeps `build_session_context` **byte-identical** and adds a parallel display derivation â€” observably equivalent to pi on both tiers, mechanism documented here:

- **`session/context.py`:** `CustomMessage` (frozen, `role="custom"`, pi messages.ts:46-53 shape) + `create_display_custom_message` (pi `createCustomMessage` faithful) + `select_display_entries` â€” the compaction-boundary survivor selection EXTRACTED from `build_session_context` (chosen compaction leads as the summary marker; non-chosen compaction entries filtered; behavior-identical refactor) â€” + `build_display_messages` = same boundary, same summary wrapping, but `custom_message` stays rich. Display output must never feed the LLM pipeline.
- **`ExtensionRunner.get_message_renderer(custom_type)`** (agent-core): pi runner.ts:546-554 mirror â€” first-wins in load order, read live per call, duck-typed, **no collision warning** (pi has none; contrast `get_shortcuts`).
- **`EventRenderer`:** `replay` gains the `role == "custom"` branch â€” the `display` gate fires BEFORE any lookup; `_render_custom` tries the late-bound `render_custom_message` hook (the `get_tool_renderer_desc` idiom) and on `None`/raise falls through **silently** (pi custom-message.ts:68-70) to the default: bold `[custom_type]` label + plain content text (dict-or-object content is read tolerantly â€” the wire form is raw dicts, issue #62 review). **Divergences from pi's default (custom-message.ts:78-97), all documented not matched:** pi draws a themed box with a leading blank spacer and a theme-accent label color, and renders content as markdown, and re-invokes the renderer on expand-toggle; aelix scrollback is plain-text-first, hardcodes a `bold magenta` label with no spacer, and is static-per-replay (the thinking-toggle divergence class, ADR-0123).
- **`shell.py`:** `_render_custom_message` closure (live runner lookup â†’ `renderer(msg, MessageRenderOptions(expanded=ctx.get_tools_expanded()), ctx.theme)` â†’ `Component.render(80)` â†’ `Text.from_ansi`; never-raises) wired onto the renderer; `_display_messages(session)` feeds both replay callsites from `get_branch()` + `build_display_messages`, **degrading** to the old `build_context().messages` path when a session (test fakes, alt backends) lacks `get_branch`.
- **`extensions/api.py`:** `MessageRenderOptions` dataclass; `MessageRenderer` docstring now specifies the pi call convention (`(message, options, theme) â†’ Component | None`) â€” the de-facto public extension contract.

**Behavior changes (intended, pi-correct):** a `display=False` custom message no longer renders in replay (it used to leak as a `Â»` user echo); a `display=True` one renders under its `[custom_type]` label (or its extension renderer) instead of masquerading as user input. **Live emission stays out of scope â€” but the reason is aelix-side, not a pi absence (issue #62 review corrected an earlier misstatement):** pi *does* have a live push site â€” `sendCustomMessage` (agent-session.ts:1312) appends a `CustomMessageEntry` and drives the same `addMessageToChat` `case "custom"` render path. Aelix's equivalent send path is a Sprint-5a stub (`ExtensionAPI.send_message`, api.py) with no emit and no `custom_message` `AgentEvent` variant, so LIVE custom-message rendering is deferred on the aelix side; the harness's `PendingCustomMessageWrite` is only a defensive turn-flush arm (pi likewise does not auto-push custom messages through the turn flush â€” the explicit `sendCustomMessage` API is separate). This ADR wires the REPLAY path only.
