## Context

TUI-first v1 Sprint 1. The repo is a uv workspace and the `aelix` umbrella meta-package was already
scaffolded, but: there was **no CI**, **no PyPI metadata**, **no publish automation** (`pip install
aelix` could not work), and the **`aelix` console script collided** — the root meta defined
`aelix = "aelix.__main__:main"` (a credential-free DEMO) while `aelix-coding-agent` defined
`aelix = "aelix_coding_agent.cli.entry:main_sync"` (the REAL CLI). The user chose a **single `aelix`
meta-package** distribution and to **build the full release infra now** (the actual publish remains the
maintainer's action).

## Decision

### CI (`.github/workflows/ci.yml`)
push + pull_request → `uv sync` → `uv run ruff check .` → `uv run ruff format --check .` →
`uv run pytest -p no:cacheprovider -q` (the repo's real gate). Matrix Python 3.11 + 3.12
(`requires-python>=3.11`). `permissions: contents: read`; concurrency cancel-in-progress. All actions
SHA-pinned.

### Console-script collision fix
Removed the root meta's `[project.scripts] aelix`. `aelix-coding-agent`'s `aelix =
…cli.entry:main_sync` is now the sole owner of the `aelix` command (**verified in the built wheels**:
the `aelix` meta wheel has no `entry_points.txt`; the `aelix_coding_agent` wheel carries the real CLI
console script). The demo stays reachable via `python -m aelix`.

### Packaging metadata (4 published packages: aelix, aelix-ai, aelix-agent-core, aelix-coding-agent)
Added `classifiers` (Beta, Apache license, Python 3.11/3.12, AI/Frameworks topics), `keywords`, and
`[project.urls]` (Homepage/Repository/Issues → github.com/handochan/aelix-ai); per-package READMEs.
**Inter-package deps pinned `==0.1.0`** so the published umbrella resolves a lock-step set (**verified**:
the meta wheel METADATA is `Requires-Dist: aelix-ai==0.1.0` / `aelix-agent-core==0.1.0` /
`aelix-coding-agent==0.1.0`). The dev-only `[tool.uv.sources] workspace=true` entries are stripped at
build.

### Release (`.github/workflows/release.yml`)
On a `v[0-9]+.[0-9]+.[0-9]+*` tag: a strict-semver tag-regex gate → `uv build --all-packages` → drop
the `aelix-server` artifacts → publish via **PyPI Trusted Publishing** (OIDC; `id-token: write` only;
**no API token stored**) behind a `pypi` GitHub Environment gate, with **PEP 740 build provenance
attestations**. `aelix-server` (the deferred Web-UI daemon) is **excluded** from the v1 publish set.

### CHANGELOG.md + RELEASING.md
Keep-a-Changelog `CHANGELOG.md` (0.1.0 initial). `RELEASING.md` documents the release process and the
**one-time maintainer setup** (create the 4 PyPI projects' Trusted Publishers / pending publishers
pointing at this repo + `release.yml` + the `pypi` environment).

## Security posture (publish pipeline)

All 5 third-party actions SHA-pinned (each verified against the GitHub API to match its labelled tag);
Trusted Publishing (no long-lived token in the repo); minimal per-job permissions; the publish trigger
requires repo write access (tag push) + the strict-semver gate + the environment approval; build
provenance attestations. Implemented + 3-lens review (security / packaging / ci-validity) — 9 findings,
**0 non-LOW**; the 2 LOW security hardenings (tighten the `v*` glob + add a regex gate; explicit PEP 740
attestations) were applied on top.

## NOT done (intentional)

- **The actual PyPI publish** — outward-facing; the maintainer configures Trusted Publishers + pushes a
  release tag. No publish was performed.
- **A logo / branding kit** (`docs/assets/`) was generated by an implementer but **held back from this
  commit** — branding is a product/owner decision, and a 530KB AI-generated logo should not be adopted
  as the official mark without the owner's choice. Available to add on request.
