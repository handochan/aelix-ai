# ADR-0181 — #21/#20 W1: declarative manifest activation (mcp_servers + lazy on_command) + extension shortcuts

- **Status:** Accepted — **LIVE**.
- **Date:** 2026-07-03
- **Sprint:** Declarative extension surface, wave 1 (owner-approved scope 2026-07-03).
- **Pi pin:** `earendil-works/pi@734e08e`. **aelix-original** — pi has NO capability manifest / activation events (verified: pi's `pi` manifest in package.json declares only WHICH FILES load; all capability registration is imperative `register*`). Design is VS Code-inspired per ADR-0096.
- **Relates:** ADR-0096 (manifest v1 + "lazy load is mandatory" activation policy), ADR-0098 (foundation lock), ADR-0102 (Tier-4b subprocess hooks — the only previously-wired contributes family), ADR-0170 (CommandDispatchService), ADR-0177/0179 (#24 reload — the live-read idioms reused here). GitHub #21 (partial), #20 (shortcuts half).

## Context

The manifest schema (`aelix-plugin.toml`) declares `contributes.{commands,tui_widgets,descriptors,tools,themes,mcp_servers,hooks}` plus `activation.{on_startup_finished,on_command,on_tool_call,on_session_start}` — but only `contributes.hooks` was wired; everything else was Pydantic-validated then dropped, and `activation` (a REQUIRED field) had zero runtime consumers. Recon readiness per family: `mcp_servers` = drop-in (the manager already consumes `list[McpServerContrib]`, fed only from `.aelix/mcp.json`); `commands`/`tools` = registration chains complete but manifests carry metadata only (no handler — the VS Code model: activation runs the factory, which registers imperatively); `themes`/`descriptors` = greenfield/shape-mismatch (deferred, follow-up issues); activation engine = entirely missing (everything loads eagerly). #20: `register_shortcut`/`register_message_renderer` stored but had ZERO consumers; recon proved custom messages also have NO live render path (pi has no push site either) and the replay source (`build_context()`) flattens custom entries DISCARDING `custom_type` — so even the replay-scoped renderer needs an entry-stream replay refactor (descoped to a follow-up with this evidence; owner-approved scope adjusted).

**Lazy feasibility ground truth (recon):** late (post-harness-construction) registrations reach the live harness for COMMANDS (runner iterates live), TOOLS (`refresh_tools` action exists), flags and providers — but HOOKS were construction-time-only: the HookBus handler map is snapshotted once and there was no `refresh_hooks`. Silently-dead late hooks would be a landmine, so W1 adds the symmetric refresh (protected-core, pattern-following).

## Decision

- **Slice A — `contributes.mcp_servers`:** the loader's 4-tier discovery is extracted into `_discover_entries()`; new `scan_extension_manifests()` = **metadata-only** scan (parses manifests, never imports/executes plugin code — the entry_points tier is EXCLUDED from the scan because `ep.load()` is an import). `cli/entry.py` scans before the MCP connect (which precedes the first harness build) and merges manifest servers into the connect list. Trust inherited (`no_project_local=not project_trusted`); an explicit `.aelix/mcp.json` entry WINS a name collision (manager dict is last-wins; manifest contribs are ordered so tier priority is preserved among manifests). **W1 limitation (documented):** MCP connects once at startup — manifest servers take effect on the next process start, not `/reload`.
- **Slice B — VS Code-style lazy activation (declarative-scoped):** a manifest plugin whose triggers are PURE `on_command` — and which declares no `contributes.tools` (declared tools must be visible to the model ⇒ eager) — is DEFERRED: an empty `Extension` shell (manifest attached) joins the loaded set; a `PendingActivation` parks on `_ExtensionRuntime.pending_activations`; NO module import happens. `CommandDispatchService` surfaces pending triggers as autocomplete stubs and, on invocation, activates (`activate_pending_extension`: resolve → factory against the SAME shell/runtime → declarative-hook wiring → `refresh_tools` + `refresh_hooks`), then re-resolves and executes. Activation failure or a declared-but-unregistered trigger is a user-visible ERROR (never a silent model fall-through), one-shot (no per-keystroke retry), and **rolls the shell back** (all registration surfaces cleared + refreshes re-run) so a mid-factory crash cannot leave a half-activated plugin. Eager triggers (`on_startup_finished`/`on_session_start`/`on_tool_call`) keep today's load-time factory run. `/reload` re-defers naturally (fresh runtime per rebuild).
- **`refresh_hooks` (protected core):** the bind_core hook-wiring loop became the re-runnable `AgentHarness._wire_extension_hooks()` — idempotent via `(event, id(handler))`/`id(cleanup)` seen-sets, always targeting the SAME HookBus instance (the runner bridges hold bound methods of it). Bound as the 17th runtime action (no-op pre-bind, like `refresh_tools`).
- **Slice C — #20 shortcuts:** `ExtensionRunner.get_shortcuts()` (live aggregation, first-registration-wins). `AelixChrome(extension_shortcuts=provider)`: key SPECS enumerate once at KeyBindings build; HANDLERS resolve through the live provider at fire time (reload-safe swaps; NEW keys post-reload bind on next TUI start — documented, and a lazily-activated plugin's shortcuts warn at activation). Key-spec translator (`ctrl+y`→`c-y`, `shift+tab`→`s-tab`, `alt/meta+x`→`escape,x`; native names pass through; bare printable single chars REJECTED — they would hijack typing). **Built-ins win** — collision check canonicalizes through prompt-toolkit KEY_ALIASES (`enter`≡`c-m`, `tab`≡`c-i`) so aliases can't bypass it. Extension bindings carry the same focus/modal filter as `s-tab` (no firing over the permission dialog); async handler tasks are tracked with a done-callback (no unretrieved-exception noise); every skip/failure logs, nothing raises.

## Adversarial review (4 lenses → per-finding skeptic verify: 20 findings)

All lenses' substantive findings CONFIRMED and fixed in the same pass, including: **HIGH** key-alias bypass of built-ins-win (an extension spec `enter` shadowed the core Enter submit — live-reproduced; fixed via KEY_ALIASES canonicalization), **HIGH** half-activated plugin on factory failure (live-reproduced; fixed via shell rollback + refresh purge), **MEDIUM** the scan's entry_points tier imported code (violating metadata-only; the tier is now excluded from scans — also self-found pre-review), **MEDIUM** modal-filter gap, **MEDIUM** unretrieved async-handler exceptions, **MEDIUM** `contributes.tools` plugins must stay eager, **MEDIUM** test #14 silently weakened (now asserts deferral explicitly), plus LOW/NIT: manifest-vs-manifest collision precedence, scan-error logging, printable-char hijack, vacuous test assertion, stub dedup, sys.path hygiene, id-reuse rationale documented.

## Consequences

- The declarative path is ALIVE: a manifest can now contribute MCP servers, and an `on_command` plugin costs ZERO startup work until first use — `activation` stops being dead-weight. The imperative `register_*` path is untouched; existing imperative extensions see no behavior change.
- `ExtensionRuntimeActions` is now 17 fields (`refresh_hooks` added — all constructors updated).
- **Deferred with evidence:** #20 renderer half (needs an entry-stream replay refactor — `build_context()` discards `custom_type`; live push site absent in pi too); `contributes.{themes,descriptors,tui_widgets}`; lazy `on_tool_call` (needs schema-carrying ToolContribs); manifest-MCP on `/reload`.
